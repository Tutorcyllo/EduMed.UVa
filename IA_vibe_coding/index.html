Volver a IA en EduMed

Vibe coding:
crear recursos digitales sin saber programar

Tienes una idea para una actividad interactiva, un simulador o un juego de preguntas para tus estudiantes. Pero no sabes programar. El vibe coding es la forma de convertir esa idea en una web real, dialogando con la IA.

No se trata de aprender a programar.
Se trata de saber explicar bien lo que quieres.

El profesor diseña la idea docente, describe qué quiere conseguir y para quién. La IA genera el código. Ambos lo mejoran mediante conversación — exactamente igual que cuando le pides mejoras a un prompt. El resultado: una herramienta web funcional que puedes compartir con tus estudiantes.

El nombre

¿Por qué se llama vibe coding?

🎵

De la intuición al resultado

El término lo acuñó en 2025 Andrej Karpathy, uno de los fundadores de OpenAI. Vibe en inglés significa "ambiente" o "sensación": la idea es que el programador ya no escribe instrucciones técnicas precisas, sino que describe el resultado que quiere conseguir — el ambiente, la sensación, la experiencia — y deja que la IA resuelva los detalles técnicos. Es programar guiándose por la intuición y el objetivo, no por el código.

Para los profesores, es exactamente lo mismo: describes cómo quieres que sea la actividad, qué debe sentir el estudiante al usarla, qué debe aprender. La IA hace el resto.

Contexto

¿En qué se diferencia del no-code?

Hay tres formas de crear sin escribir código. El vibe coding es la más flexible — y la que mejor se adapta a necesidades docentes concretas.

No-code

Arrastrar y soltar

Plantillas y bloques predefinidos. Rápido, pero limitado a lo que la herramienta permite.

Low-code

Visual + algo de código

Más flexible, pero requiere conocimientos técnicos mínimos para las partes personalizadas.

Vibe coding

Conversación con la IA

Describes lo que necesitas, la IA lo construye. Cualquier cosa ajustada a tu asignatura exacta.

"El profesor diseña la idea. La IA genera el código. Ambos lo mejoran hablando."

La barrera que desaparece

¿Por qué ahora sí es posible?

Muchos docentes tienen ideas muy claras sobre qué herramienta les sería útil — un simulador de decisiones, un caso clínico interactivo, un cuestionario con retroalimentación inmediata — pero hasta ahora convertirla en una web requería saber programar. Ya no.

Lo que sí sigue siendo tuyo, y lo más importante, es la idea docente: saber qué quieres que aprendan los estudiantes, cómo quieres que lo practiquen y qué debe ocurrir cuando se equivocan. Eso no lo genera la IA. Lo pones tú.

El vibe coding no convierte al profesor en programador. Lo convierte en diseñador de experiencias educativas digitales. La clave está en saber formular bien la necesidad, no en entender el código.

Referencia

Palabras técnicas que conviene conocer

No hace falta saber programar, pero sí conviene reconocer estos términos cuando aparezcan — en el prompt, en la respuesta de la IA o en un tutorial.

HTML

HTML

El formato básico de las páginas web

HTML es el lenguaje que describe la estructura de una página web: qué es un título, qué es un párrafo, dónde va un botón. No es un lenguaje de programación — es más parecido a un documento de Word con etiquetas invisibles que organizan el contenido.

Cuando la IA genera "un archivo HTML", está creando un documento que cualquier navegador (Chrome, Safari, Firefox) puede abrir y mostrar como una web.

UX

Frontend, Backend, UX y UI

Las cuatro caras de una aplicación web

Frontend es todo lo que el estudiante ve e interactúa: botones, textos, colores, pantallas. Es la "fachada" de la aplicación. El vibe coding trabaja casi siempre en el frontend.

Backend es lo que ocurre detrás, en el servidor: guardar datos, gestionar usuarios, conectar servicios. Para un primer recurso docente, no hace falta backend.

UX (User Experience, experiencia de usuario) se refiere a si la aplicación es fácil, clara y agradable de usar. Una buena UX significa que el estudiante entiende qué hacer sin instrucciones.

UI (User Interface, interfaz de usuario) es la parte visual: colores, tipografía, tamaño de botones, organización de la pantalla. Una buena UI hace que la aplicación se vea bien y sea coherente.

API

API

La conexión entre aplicaciones

Una API es el sistema que permite que dos aplicaciones se comuniquen. Cuando una web guarda resultados en Google Sheets, o cuando muestra un mapa de Google Maps, está usando una API: le pregunta a otro servicio y recibe una respuesta.

Para un primer recurso docente sencillo no necesitas APIs. Aparecen cuando el recurso necesita conectarse con servicios externos — guardar datos, enviar correos, mostrar información en tiempo real.

🐛

Depuración (debugging)

El proceso de encontrar y corregir errores

Depurar (en inglés, debug) significa encontrar y corregir errores en el código. Es inevitable: casi ningún recurso funciona perfectamente desde el primer intento.

En vibe coding, depurar es sencillo: describes el error a la IA en lenguaje normal y ella lo corrige. Por ejemplo: "Al pulsar el botón 'siguiente', la pantalla se queda en blanco en lugar de avanzar al paso 2." Cuanto más concreta sea la descripción del fallo, más rápido lo resuelve.

🔑

Autenticación y HTTPS

Seguridad y acceso a la aplicación

Autenticación es el sistema que verifica quién es el usuario: el clásico nombre de usuario y contraseña. Añadirla a un recurso lo complica mucho y abre problemas de privacidad y gestión. Para empezar, evítala.

HTTPS es el protocolo de seguridad que protege la comunicación entre el navegador del estudiante y el servidor donde está alojada la web. Se identifica por el candado en la barra del navegador. Es importante si el recurso maneja cualquier tipo de datos personales — aunque para recursos educativos sencillos sin datos, no es crítico.

Qué puedes crear:
recursos web para la docencia médica

El formato más útil para empezar es siempre una web app — una página que se abre en el navegador, se puede compartir con un enlace e incrustar en Moodle. Sin instalación, sin cuentas.

Caso clínico ramificado
El estudiante elige preguntas, pruebas y tratamientos — y el caso evoluciona según sus decisiones

Muchos estudiantes aprenden a describir enfermedades pero no a razonar ante un paciente real. El caso ramificado obliga a pensar en secuencia, igual que en la clínica: con información incompleta, incertidumbre y consecuencias.

Ejemplos médicos
  • Diagnóstico diferencial de disnea aguda con tres vías posibles
  • Manejo inicial del dolor torácico en urgencias
  • Paciente con fiebre, soplo y hemocultivos positivos
  • Anemia microcítica: ¿ferropénica, talasemia o inflamatoria?
Qué construiría el profesor

Un caso por pasos con botones de decisión, retroalimentación inmediata tras cada elección y una explicación del razonamiento experto al final. El estudiante puede repetirlo eligiendo otras opciones.

🎛️
Simulador de "¿qué pasaría si…?"
El estudiante modifica variables y observa cómo cambia un proceso fisiológico

La fisiología se suele enseñar como hechos fijos. Un simulador permite descubrir relaciones causales: ¿qué ocurre si baja la albúmina? ¿Y si cae el filtrado glomerular? El estudiante experimenta con el sistema en lugar de memorizar sus respuestas.

Ejemplos médicos
  • Cambios hemodinámicos en distintos tipos de shock
  • Relación entre albúmina, presión oncótica y edema
  • Efecto de la pérdida de nefronas sobre creatinina y filtrado
  • Equilibrio ácido-base al modificar ventilación o bicarbonato
Qué construiría el profesor

Una interfaz con deslizadores, gráficos simples y mensajes interpretativos que cambian en tiempo real. Por ejemplo: "al reducir la albúmina, disminuye la presión oncótica y aumenta la tendencia al edema".

🎮
Juego de preguntas
Repaso gamificado con puntuación, límite de tiempo y explicación de cada opción

El repaso pasivo (releer apuntes) es poco eficaz. Un juego de preguntas obliga a recuperar activamente la información, detecta lagunas concretas y mantiene la motivación mediante la puntuación y el reto.

Ejemplos médicos
  • Preguntas tipo MIR con explicación de cada opción incorrecta
  • Identificación de patrones analíticos: ¿qué diagnóstico corresponde?
  • Clasificación de fármacos por mecanismo de acción
  • Reconocimiento de signos clínicos por descripción
Qué construiría el profesor

Una serie de preguntas con cuatro opciones, temporizador, puntuación acumulada, retroalimentación inmediata y resumen final con los errores cometidos y sus explicaciones.

Recursos para consolidar conceptos de forma activa. El estudiante controla el ritmo, puede repetir cuantas veces necesite y detecta por sí mismo sus lagunas.
🃏
Flashcards web
Tarjetas de repaso interactivas con respuesta oculta y autoevaluación

Las flashcards en papel exigen tiempo de preparación y no se pueden compartir fácilmente. Una versión web permite al profesor crear el mazo una vez y distribuirlo con un enlace; el estudiante las repasa en el móvil cuando quiere.

Ejemplos médicos
  • Signos clínicos de enfermedades autoinmunes por órgano
  • Antibióticos: espectro, mecanismo y uso principal
  • Autoanticuerpos y su enfermedad asociada
  • Efectos adversos de fármacos de uso frecuente
Qué construiría el profesor

Tarjetas con pregunta en el anverso, respuesta en el reverso (que gira al pulsar), botones para marcarla como "dominada" o "repasar de nuevo" y un contador de progreso.

📖
Glosario interactivo
Términos clave con definición, ejemplo clínico y búsqueda instantánea

Los glosarios en PDF son difíciles de consultar sobre la marcha. Una versión web con búsqueda instantánea permite al estudiante encontrar un término en segundos, con su definición y un ejemplo clínico.

Ejemplos médicos
  • Glosario de términos de fisiopatología con ejemplo clínico
  • Términos de farmacología: farmacocinética y farmacodinámica
  • Glosario de IA en medicina para profesores
  • Conceptos de epidemiología clínica y estadística básica
Qué construiría el profesor

Una página con buscador en tiempo real, listado alfabético y tarjetas desplegables por término con definición clara, ejemplo en contexto clínico y términos relacionados.

🏆
Página gamificada
Apuntes convertidos en una experiencia progresiva con niveles, puntos y retos

Leer apuntes linealmente no genera compromiso. La gamificación añade estructura, metas visibles y recompensas inmediatas que mantienen la motivación, especialmente útil antes de un examen.

Ejemplos médicos
  • Repaso de farmacología con niveles desbloqueables por tema
  • Preparación para el examen MIR con puntuación acumulada
  • Recorrido gamificado por los órganos del sistema inmune
  • Misión diagnóstica: resolver casos para avanzar de nivel
Qué construiría el profesor

Una secuencia de bloques temáticos que se desbloquean al superar mini-pruebas, con barra de progreso, puntuación, badges por tema completado y pantalla de resultados final.

Hay contenidos médicos que se entienden mejor cuando dejan de ser una lista de conceptos y se convierten en una imagen manipulable. Para hacer visible lo que en una diapositiva estática queda oculto.
🗺️
Mapa conceptual interactivo
Una red de conceptos clicables que el estudiante explora por capas: visión general primero, mecanismos después

En temas complejos, el estudiante se pierde entre mediadores, órganos, manifestaciones clínicas y pruebas diagnósticas. Un mapa interactivo permite explorar por capas, evitando la sobrecarga de información.

Ejemplos médicos
  • Cascada inflamatoria en una enfermedad autoinmune
  • Fisiopatología del shock séptico, cardiogénico o hipovolémico
  • Relación entre obesidad, insulinorresistencia e inflamación
  • Manifestaciones del lupus eritematoso sistémico por órganos
Qué construiría el profesor

Nodos conectados con colores por sistemas, ventanas emergentes con explicaciones breves al pulsar, y botones para mostrar niveles de complejidad: básico, intermedio y avanzado.

⏳
Línea temporal de enfermedad
La enfermedad como película, no como fotografía: fase a fase, con síntomas, decisiones y puntos de no retorno

Los estudiantes suelen estudiar las enfermedades como fotografías aisladas, cuando muchas se comprenden mejor como películas. Una línea temporal permite ver qué aparece primero, qué datos son tardíos y cuándo conviene intervenir.

Ejemplos médicos
  • Historia natural de la infección por VIH sin tratamiento
  • Fases de la sepsis: infección local → respuesta sistémica → shock
  • Evolución de la hepatitis B desde la infección aguda hasta la cronicidad
  • Curso de una enfermedad autoinmune hasta daño irreversible
Qué construiría el profesor

Una línea con fases clicables, tarjetas explicativas, marcadores de "punto de no retorno", alertas clínicas y preguntas del tipo: "¿Qué dato esperarías en esta fase?"

🧮
Calculadora docente paso a paso
No solo da el resultado: muestra cómo se calcula, qué significa y qué decisión clínica implica

Los estudiantes suelen memorizar fórmulas sin entenderlas. Una calculadora docente muestra el cálculo desglosado, la categoría de riesgo y cómo pequeños cambios en una variable modifican la decisión final.

Ejemplos médicos
  • Filtrado glomerular estimado con interpretación por estadio
  • Escala CURB-65 en neumonía: riesgo y decisión de ingreso
  • CHA₂DS₂-VASc y HAS-BLED en fibrilación auricular
  • Interpretación docente de gasometría arterial básica
Qué construiría el profesor

Un formulario con los datos clínicos necesarios, cálculo desglosado paso a paso, interpretación final con categoría de riesgo y un aviso: "esta herramienta sirve para aprender el razonamiento, no para sustituir la valoración clínica".

🧩
Comparador visual de diagnósticos
Selecciona dos o tres enfermedades y compara síntomas, pruebas clave y errores frecuentes lado a lado

El diagnóstico diferencial es una de las competencias más difíciles de enseñar. Un comparador visual obliga al estudiante a mirar las enfermedades como alternativas que compiten, igual que ante un paciente real.

Ejemplos médicos
  • Artritis reumatoide, lupus y espondiloartritis
  • Disnea: insuficiencia cardiaca, EPOC, neumonía o TEP
  • Meningitis bacteriana, vírica y encefalitis
  • Síndrome nefrótico frente a síndrome nefrítico
Qué construiría el profesor

Una tabla interactiva donde el estudiante selecciona dos o tres diagnósticos y ve sus diferencias en columnas paralelas, con colores para los datos discriminativos y una sección de "pistas clave".

🧬
Atlas interactivo de mecanismos
Capas activables de células, mediadores, órganos y fármacos para entender sistemas complejos

Hay contenidos con muchos elementos simultáneos — inmunología, coagulación, farmacología — que resultan casi imposibles de explicar solo con texto. Un atlas interactivo permite activar y desactivar capas para entender cómo se conectan los componentes.

Ejemplos médicos
  • Activación del complemento por vías clásica y alternativa
  • Sinapsis inmunológica y activación linfocitaria
  • Mecanismo de acción de fármacos biológicos
  • Coagulación e inflamación en la sepsis
Qué construiría el profesor

Una imagen central con capas activables mediante botones: mostrar células, añadir mediadores, revelar órganos diana, marcar puntos de acción farmacológica. Cada capa incluye una explicación breve.

Transformar contenido denso — un artículo, unos datos, una clase compleja — en algo visualmente accesible, navegable y fácil de compartir con estudiantes o colegas.
📄
Landing page de artículo científico
Una guía clínica o artículo transformado en infografía navegable con conclusiones e implicaciones docentes

Los artículos científicos son densos y difíciles de leer para estudiantes de grado. Una landing page convierte el contenido esencial en algo visual, accesible y con contexto docente: qué importa de este artículo y qué implica para la práctica.

Ejemplos médicos
  • Guía clínica de insuficiencia cardiaca resumida visualmente
  • Estudio clave de farmacología con resultados principales
  • Protocolo de actuación en urgencias en formato visual interactivo
  • Meta-análisis con gráfico de forest plot explicado
Qué construiría el profesor

Una página con resumen visual, secciones desplegables (método, resultados, limitaciones), mensajes clave destacados y una sección "¿Qué cambia en mi práctica?" para anclar el aprendizaje.

📊
Dashboard de resultados de prácticas
Datos de una encuesta o práctica convertidos en gráficos interactivos que los estudiantes pueden explorar

Mostrar datos de prácticas en una tabla de Excel no genera reflexión. Un dashboard interactivo permite a los estudiantes explorar los resultados, comparar grupos y sacar sus propias conclusiones antes de la discusión en clase.

Ejemplos médicos
  • Resultados de una encuesta de percepción del riesgo cardiovascular
  • Comparativa de respuestas antes y después de una intervención
  • Datos epidemiológicos de una práctica de campo
  • Resultados de autoevaluación diagnóstica de los estudiantes
Qué construiría el profesor

Una página con gráficos de barras o tartas, filtros por variable, cifras clave destacadas y preguntas de reflexión que guían la interpretación de los datos.

🎤
Guion interactivo para exposiciones
Apoyo visual para presentaciones orales con notas del ponente, temporizador y transiciones controladas

Las presentaciones en PowerPoint no siempre se adaptan bien a defensas de TFG o comunicaciones en jornadas. Un guion interactivo web permite al ponente ver sus notas en el móvil mientras la audiencia ve la presentación en pantalla.

Ejemplos médicos
  • Guion para la defensa del TFG con puntos clave por sección
  • Presentación de un caso clínico con notas del ponente
  • Comunicación en jornada con temporizador y alertas de tiempo
  • Exposición de seminario con preguntas anticipadas
Qué construiría el profesor

Una página con diapositivas navegables, notas del ponente en panel lateral, temporizador por sección, avance con teclas o botones y modo presentación a pantalla completa.

Cómo se hace:
cuatro fases y una conversación

El proceso no empieza abriendo la IA. Empieza pensando bien la necesidad docente. Cuanto más clara tengas la idea, mejor será el primer resultado — y menos vueltas habrá que dar.

1
🧠

Pensar la idea antes de abrir la IA

El único paso que hace el profesor solo

Una mala idea docente no mejora por tener una web bonita. Antes de escribir nada, responde estas preguntas:

¿Qué problema docente quiero resolver? ¿A qué estudiantes va dirigido? ¿Qué harán los estudiantes con esto? ¿Qué aprenderán al usarlo? ¿Es para clase, Moodle o estudio autónomo? ¿Necesita guardar datos?

También conviene imaginar cómo será: pantalla inicial, botones, secciones, mensajes de error. Incluso un boceto en papel o una diapositiva que puedas subir a la IA como referencia visual.

2
✍️

Describir la funcionalidad con un buen prompt

Cuanto más concreta la descripción, mejor el primer resultado

No basta con decir "hazme una web sobre lupus". La IA necesita saber para quién es, qué debe hacer el estudiante, qué debe ocurrir en cada paso y qué restricciones tiene. Esta plantilla cubre todo eso:

Plantilla de prompt — copia y adapta
Quiero desarrollar una web app educativa sobre [tema]. Objetivo: [Qué problema docente resuelve y qué debe aprender el estudiante.] Público objetivo: [Curso, titulación, nivel de conocimientos, contexto de uso.] Tipo de recurso: [Simulador / caso clínico / juego / glosario / calculadora / otro.] Plataforma: Quiero una web app en HTML, responsive (que se adapte al móvil), que pueda abrirse en navegador y compartirse con estudiantes. Contenido: Usa exclusivamente el siguiente contenido: [Pegar aquí el contenido validado por el profesor.] Funciones principales: 1. [Función 1] 2. [Función 2] 3. [Función 3] Interfaz: - Pantalla inicial con título, objetivo y botón de inicio. - Secciones claras y botones grandes. - Diseño legible en móvil y en ordenador. Diseño visual: - Estilo [sobrio / clínico / universitario / gamificado / minimalista]. - Tipografía clara y buen contraste de colores. Datos: - No debe guardar datos personales. - Si necesita puntuación, que sea local y no se envíe a ningún servidor. Solicito: Genera una primera versión funcional en un único archivo HTML con CSS y JavaScript integrados. Después explícame brevemente cómo puedo probarla y qué partes puedo modificar.
3
🔁

Probar, detectar errores y pedir mejoras

El primer resultado es siempre un punto de partida

Prueba la primera versión como si fueras estudiante. Hazte estas preguntas:

¿Se entiende qué hay que hacer? ¿Funciona en móvil? ¿Los botones hacen lo esperado? ¿Hay errores conceptuales? ¿La IA ha inventado contenido? ¿La actividad ayuda realmente a aprender?

Después pide cambios concretos. El diálogo iterativo con la IA es la esencia del vibe coding:

Ejemplo de conversación de mejora
Tú: "El texto de la primera pantalla es demasiado largo. Resúmelo en tres frases."
IA: [genera la versión resumida]
Tú: "Bien. Ahora el botón de 'siguiente' no funciona en móvil. Corrígelo."
IA: [corrige el código]
Tú: "No añadas contenido nuevo. Solo corrige lo que te indico y conserva todo lo demás."
Si la IA empieza a cambiar cosas que no le has pedido, díselo explícitamente: "Conserva todo tal como está y modifica solo lo que te señalo."
4
✅

Validar antes de usar con estudiantes

Formativo primero, certificativo después — si acaso

Antes de compartirlo con grupos grandes, pruébalo con una o dos personas de confianza. Valora:

¿Es claro para alguien que no lo ha visto antes? ¿El tiempo que requiere es razonable? ¿El contenido es correcto y verificado? ¿El nivel es adecuado para el curso?

Para empezar, úsalo como actividad formativa — práctica, autoevaluación, repaso — no como evaluación certificativa. Cuando el recurso ya funcione bien en ese rol, podrás plantearte darle más peso.

Antes de publicar:
precauciones y formas de compartir

El recurso funciona bien en tu pantalla. Antes de compartirlo, conviene revisar estos cinco puntos — especialmente si trabajas en un entorno de salud donde los datos sensibles son una responsabilidad real.

Precauciones

Cinco cosas que revisar siempre

🔒

Sin datos sensibles

No incluyas datos identificativos de estudiantes, historiales clínicos reales ni calificaciones oficiales — ni en el recurso que creas, ni en el prompt que le das a la IA para generarlo.

Si el recurso necesita que el estudiante introduzca su nombre, usa un alias o simplemente no lo pidas. En el contexto sanitario y educativo, la protección de datos no es opcional.

🔎

Revisa siempre el contenido

La IA puede inventar datos, simplificar en exceso o cometer errores clínicos con total apariencia de precisión. Todo el contenido médico debe ser revisado por el profesor antes de publicar el recurso.

Esto no es una precaución menor. Un error en una dosis, un diagnóstico incorrecto o una referencia inventada puede llegar a cientos de estudiantes en minutos.

🗄️

Empieza sin bases de datos

Una base de datos es el sistema que guarda información de forma permanente en un servidor: respuestas de estudiantes, puntuaciones, registros de uso. Añadirla a un recurso web implica diseñar cómo se almacena, gestionar quién puede acceder y asumir responsabilidades legales sobre esos datos.

Para un primer recurso, prescinde de ello. Si necesitas puntuación, que sea local — visible para el estudiante en pantalla pero sin guardarse en ningún servidor.

📱

Pide siempre diseño responsive

Responsive significa que la web se adapta automáticamente al tamaño de la pantalla: se ve bien tanto en un ordenador de sobremesa como en una tableta o un móvil. No es un ajuste estético — es funcional. Una web no responsive en un móvil puede tener texto diminuto, botones imposibles de pulsar o contenido que se sale de la pantalla.

Los estudiantes usarán el recurso mayoritariamente desde el móvil. Inclúyelo siempre en el prompt: "diseño responsive, legible en móvil".

Cuida la accesibilidad

Un recurso accesible es utilizable por el mayor número de personas posible, incluyendo quienes tienen dificultades visuales, motoras o cognitivas. No requiere grandes esfuerzos si se pide desde el principio.

Incluye en el prompt: buen contraste entre texto y fondo, texto legible (tamaño suficiente, fuente clara), botones grandes, y que la información nunca dependa únicamente del color para transmitirse.

Distribución

Formas de compartir el recurso

Más fácil

Enlace desde la plataforma

La opción más rápida. Claude, ChatGPT y Gemini permiten compartir el artifact con un enlace. Sin descargas ni instalaciones.

Fácil

Descargar el HTML

Copia el código, pégalo en el Bloc de notas, guárdalo como index.html y ábrelo con el navegador. Se puede compartir como archivo adjunto.

Medio

Incrustar en Moodle

Sube el archivo HTML como recurso o incrusta el enlace externo. La configuración institucional de Moodle puede limitar algunos elementos.

Un paso más

Publicar en GitHub Pages

Gratuito, con URL estable y sin dependencia de plataformas externas. Requiere aprender unos pasos iniciales, pero permite una colección permanente de recursos.

"El vibe coding no convierte al profesor en programador. Lo convierte en diseñador de experiencias educativas digitales."

La clave no es entender el código. Es saber formular bien la necesidad, probar con criterio docente, detectar errores de contenido y decidir cuándo una herramienta aporta valor real al aprendizaje.
